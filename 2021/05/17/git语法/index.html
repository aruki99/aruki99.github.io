创建新节点

git commit;

创建分支名字

git branch 分支名字;

选择分支

git checkout 分支名字;

合并分支

git merge 分支名字;

​ 例子：

​ git checkout 1；

​ git merge 2;

​ 注释：选着分支1，将分支2合并到分支1中。

移动分支

git rebase 分支名字;

例子：

​ git checkout 1；

​ git rebase 2;

注释：选着分支1，将分支1移动到分支2前

分离HEAD

原本

HEAD -> main -> C1

执行 git checkout C1

结果

HEAD -> C1

相对引用~num / ^ 移动HEAD

git checkout bugFix^

移到bugFix的第一个父节点

git checkout bugFix^^^

git checkout bugFix~3

移到bugFix的第3个父节点

强制制动节点位置

git branch -f main C6

将main强制移动到C6

git reset

git reset HEDA^ / git reset HEDA~1

撤回到上次操作的位置、本地看不到被撤销的操作。

想象成“改写历史”。git reset 向上移动分支，原来指向的提交记录就跟从来没有提交过一样。

git revert(远程分支无效)

在我们要撤销的提交记录后面居然多了一个新提交！这是因为新提交记录 C2' 引入了更改 —— 这些更改刚好是用来撤销 C2 这个提交的。也就是说 C2' 的状态与 C1是相同的。

revert 之后就可以把你的更改推送到远程仓库与别人分享啦。

移动分支cherry-pick

将 side 分支上的工作复制到 main分支

执行 git cherry-pick C2 C4

提交记录 C2 和 C4， C2、C4被抓到当前分支下

可视化-移动节点

git rebase -i main~4

在main节点开始数向上4个节点

可以进行删除、移动等操作

修改上次提交

git commit –amend

有时你提交过代码之后，发现一个地方改错了，你下次提交时不想保留上一次的记录；

远程

克隆

git clone

本地仓库会多出一个 origin分支

origin为远程仓库默认命名

如果

执行： git checkout origin/main; git commit

会出现HEAD分离状态

获取数据

git fetch

git fetch 通常通过互联网（使用 http:// 或 git:// 协议) 与远程仓库通信。

git fetch 并不会改变你本地仓库的状态。它不会更新你的 main 分支，也不会修改你磁盘上的文件。

理解这一点很重要，因为许多开发人员误以为执行了 git fetch 以后，他们本地仓库就与远程仓库同步了。它可能已经将进行这一操作所需的所有数据都下载了下来，但是并没有修改你本地的文件。我们在后面的课程中将会讲解能完成该操作的命令 :D

所以, 你可以将 git fetch 的理解为单纯的下载操作。

远程信息合并本地

git pull 就是 git fetch 和 git merge 的缩写！

本地上传数据

git push
